3. Understandable

What does “Understandable” mean?

In WCAG 2.2, “Understandable” means that students can understand your content and how your course works. Instructions, navigation, and feedback should be clear, consistent, and not confusing.

For CSU faculty, this often means:

  • Using student-friendly language and explaining specialized terms.
  • Keeping your course layout and navigation predictable week to week.
  • Giving clear instructions and useful error messages in quizzes and forms.
  • Offering consistent ways for students to get help or ask questions.

Section 3 – Understandable (WCAG 2.2)

3.1

Readable

Show details

WCAG Guideline 3.1 (Readable) focuses on making text clear and understandable, and correctly identifying the language of the page and any changes in language.

What this means:

Students shouldn’t have to guess what you mean. The main language should be set correctly, and jargon, abbreviations, and unusual words should be explained so all students can follow along.

Key requirements for your course:

  • Write instructions and assignment prompts in plain, direct language where possible.
  • Explain discipline-specific terms, acronyms, and abbreviations the first time you use them.
  • Set the correct document language (e.g., English) in Word/PDF so screen readers use the right pronunciation.
  • When using another language (e.g., Latin terms, Spanish quotes), provide translations or context.
  • Break up long paragraphs with headings, bullets, and white space to support reading comprehension.
Example to analyze:
An assignment prompt states: “Demonstrate metacognitive strategies for intradisciplinary synthesis.”
Better approach:
“Show how you think about your own learning as you connect ideas from different topics in this course. In your reflection, describe: (1) what you already knew, (2) what you learned, and (3) how these ideas fit together.”

3.2

Predictable

Show details

WCAG Guideline 3.2 (Predictable) requires that pages and components appear and operate in consistent, expected ways.

What this means:

Your Brightspace course should behave the way students expect: same structure each week, no sudden jumps in context, and help information in predictable locations.

Key requirements for your course:

  • Use a consistent weekly/module pattern (e.g., Overview → Materials → Activities → Assessments).
  • Avoid unexpected changes of context when students move focus or enter data (e.g., don’t auto-submit on focus changes).
  • Use consistent navigation menus and labels across your courses when possible.
  • Use descriptive, predictable labels for buttons and links (e.g., “Start Quiz 1” instead of “Go”).
  • New in WCAG 2.2 – Consistent Help: If you provide help (office hours, support links, FAQs), place these in the same location and with the same label across your course pages.
Example to analyze:
In Week 1, the syllabus is under “Content → Week 1,” but in Week 2, it moves to a different module and the “Help” information appears only in the announcements.
Better approach:
Keep the syllabus in a clearly labeled, permanent location (e.g., “Start Here → Syllabus”) and add a small, consistent “Need Help?” block in each weekly overview with your contact info, office hours, and key support links.

3.3

Input Assistance

Show details

WCAG Guideline 3.3 (Input Assistance) requires that users receive help avoiding and correcting mistakes when they enter information.

What this means:

When students complete quizzes, forms, or assignment submissions, they should know exactly what is required, and any errors should be clearly identified with guidance on how to fix them.

Key requirements for your course:

  • Provide clear instructions at the start of quizzes, forms, and major assignments.
  • Label all input fields clearly (e.g., “CSU ID,” “Short Answer Response,” “File Upload”).
  • Mark required fields in the label or instructions (and explain what happens if they are left blank).
  • Write error messages that explain what went wrong and how to correct it, rather than just “Invalid.”
  • For high-stakes tasks (e.g., major project uploads), allow students to review entries before final submission.
  • Reduce redundant entry where possible (for example, avoid having students type the same info multiple times across course forms).
Example to analyze:
A Brightspace quiz asks for a student’s name and CSU ID in multiple places, and when they forget one, it just says “Error” with no further detail.
Better approach:
Ask for name and CSU ID only once, label the fields clearly, and if there is an error, use messages such as: “Please enter your CSU ID. It should be 8 digits long.” For graded work, provide confirmation screens or summaries before final submission.
